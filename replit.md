# TaskRankr
TaskRankr is a multi-user, offline-first task manager with hierarchical tasks, status workflows, and customizable rank fields.

## Run & Operate
- **Run Dev Server**: `npm run dev`
- **Build**: `npm run build`
- **Typecheck**: `npm run typecheck`
- **Codegen (Drizzle Kit)**: `npm run generate:drizzle`
- **DB Push**: `npm run db:push`
- **E2E Tests (User Mode)**: `npm run cy:run:user`
- **E2E Tests (Guest Mode)**: `npm run cy:run:guest`

## Stack
- **Frontend**: React, shadcn/ui, Tailwind CSS
- **Backend**: Node.js, Express
- **Runtime**: Node.js (latest LTS)
- **ORM**: Drizzle ORM
- **Validation**: Zod
- **Build Tool**: Vite
- **API**: `ts-rest`
- **State Management**: Custom React Context Providers (offline-first with background sync)

## Where things live
- **UI Components**: `client/src/components/`
- **Pages**: `client/src/pages/`
- **React Hooks**: `client/src/hooks/`
- **Database Schema**: `shared/schema/`

## Architecture decisions
- **Local-first writes + background sync**: Task mutations update local state immediately and are queued for background synchronization. No `useMutation` or `isPending` states in app code.
- **Two-context provider pattern**: Providers like `TasksProvider` expose separate contexts for reactive data (e.g., `useTasks`) and stable mutator functions (e.g., `useTaskMutations`) to optimize re-renders.
- **Dialog-scoped draft sessions**: Task form draft state is isolated within `DraftSessionProvider` (mounted inside `TaskFormDialogProvider`) to prevent re-rendering the main task list during draft edits.
- **Stable React identity for tasks**: `clientKey` (a UUID) is used as the React `key` for tasks, ensuring stable component identity across reloads and after temporary ID-to-real ID swaps during sync.
- **PWA with Workbox**: A service worker is generated by `vite-plugin-pwa` for offline capabilities and caching, configured in `vite.config.ts`.

## Product
- **Hierarchical Tasks**: Create tasks and subtasks to organize work.
- **Status Workflow**: Tasks can be `open`, `in_progress`, `pinned`, or `completed`.
- **Customizable Rank Fields**: Users can define and sort tasks by `priority`, `ease`, `enjoyment`, and `time`.
- **Offline-First**: Users can continue working without an internet connection; changes sync in the background.
- **Guest Mode**: Allows unauthenticated users to use the app with local storage and demo data.
- **Replit Auth Integration**: Secure user authentication via Replit's OIDC.
- **User Settings**: Customize attribute visibility and other preferences.
- **Progressive Web App (PWA)**: Installable for an app-like experience.

## User preferences
- Preferred communication style: Simple, everyday language.
- File naming: kebab-case for utility/helper files (e.g., `auth-utils.ts`), PascalCase for component primitives (e.g., `DropdownMenu.tsx`, `AlertDialog.tsx`), camelCase for hooks (e.g., `useAuth.ts`, `useSettings.ts`)
- Icon helper: Use `Icon` component from `LucideIcon.tsx` only for conditional/dynamic icons (ternary cases), not for single static icons
- Code comments & JSDoc: Comments and docstrings describe a function/module from the perspective of its API â€” *what* it does for callers and *why*, when the design is unique or non-obvious. Trust the code to be self-explanatory for the *how*: do not restate the implementation, avoid pasting code snippets that mirror what's right below, and do not narrate steps a reader can see. Skip hedge prefixes ("Derived predicate:", "Translates X into Y:") and lead with the claim. Don't restate what the type signature already conveys (optionality, parameter types, return types). For multi-branch behavior, prefer a one-line summary plus a tight bullet list ("including: ...") over branch-by-branch prose. Keep descriptions concise (1-2 lines for most things; longer only when there's genuine "why" to explain). Prefer renaming a variable/prop to inserting an inline comment that explains what it *is*. Use exact package names as imported (e.g., `@radix-ui` not "Radix UI").
- Terminology: "Rank fields" refers to the 4 sortable fields with badges: priority, ease, enjoyment, time (distinct from text fields like name/description)
- Test IDs: Use `data-testid` as the prop name, not `testId`
- Icon Sizing: Use `size-X` tailwind class instead of `w-X h-X`
- es-toolkit: Use `es-toolkit` helper functions when we can greatly simplify something. For example, when copying many same-named properties from one object to another, use `omit` or `pick` from `es-toolkit` instead of enumerating every field by hand. Example: `createTask({ ...omit(draft, ['id', 'userId']), parentId: resolved })` rather than listing all 14 fields explicitly.
- Formatting: Run `npm run format` (biome) before every commit/checkpoint so the code in checkpoints is always formatted.
- Documentation: Keep `replit.md` focused on cross-cutting architecture and conventions. File-level mechanics, function signatures, and internal helpers belong in the relevant file's docstring, not here.

## Gotchas
- **Cypress E2E Tests**: New test files must be manually added to `cypress.config.ts`. Always add new `data-testid` selectors to `cypress/support/constants/selectors.ts` first.
- **LocalStorage**: Always use `client/src/lib/storage.ts` for persisting user state; do not call `localStorage.*` directly.
- **Font Loading**: Only Inter and Outfit are used; do not add additional font families.
